# 小说工程第一性原理

## 1. 模型最终只吃上下文

LLM 最终只认识：

```text
system prompt
user prompt
上下文资料
tool result
历史消息
```

所以 Skill、Agent、Memory、MCP、Workflow 都不是魔法。它们的价值在于：选择什么上下文、如何压缩上下文、如何验证上下文、如何保存结果并形成下一轮上下文。

## 2. 长篇小说不能靠模型记忆

长篇小说真正难的是：

```text
第 1 章埋的伏笔，第 1000 章还能回收
第 312 章的物品细节，第 1000 章不能写错
某个角色知道什么、不知道什么，不能串台
某个爽点欠了多久，不能无限拖
某条支线离开多久，必须合理回归
```

这些不应该交给模型模糊记忆，而应该交给工程系统维护。

## 3. 原文是最终证据源

证据优先级：

```text
正式原文 > 带来源 ledger > snapshot > planning > 模型生成草稿 > 用户随口设想
```

snapshot 和 ledger 是索引、导航和状态缓存。当涉及精确细节时，必须回原文查证。

## 4. 128K 上下文不是仓库，是创作现场

128K 的目标不是装最多字，而是装最有用的信息：

```text
高密度
低噪声
强约束
有证据
有优先级
```

## 5. 写小说不是自由生成，而是状态驱动生产

标准链路：

```text
项目状态
  → 本章目标
  → 情绪任务
  → 场景卡
  → 正文生成
  → 读者体验审稿
  → 连续性审稿
  → 返修
  → snapshot / ledger 入账
```

## 6. NovelCode 的目标

不是让模型记住整本书，而是让项目结构记住小说，让模型每次只处理被编译好的高价值上下文。
